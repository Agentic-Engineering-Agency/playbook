# Caso de estudio: Billi

Billi es una app de finanzas personales diseñada para el mercado mexicano — específicamente para estudiantes universitarios de 18-25 años que quieren rastrear gastos diarios sin la complejidad de las herramientas de presupuesto tradicionales.

<Callout type="warn">
**Este caso de estudio está en proceso.** El ciclo MVP de Billi arranca el **2026-04-20** y corre por cuatro semanas hasta el **2026-05-17**. Abajo dejamos la hipótesis que estamos probando; actualizaremos esta página con los resultados observados al cierre de cada ciclo.
</Callout>

## Hipótesis que estamos probando

Una persona sin conocimientos de desarrollo puede pasar de tres documentos markdown cortos (`product-brief.md`, `ux-vibes.md`, `screens.md`) a un prototipo funcional en React la misma tarde, usando el flujo de prototype-kit descrito en este Playbook. En concreto esperamos:

- De documentación a primer render en menos de tres horas.
- Un prototipo suficientemente pulido para demostrarse frente a un jurado sin disculpas.
- Menos de una intervención manual por pantalla generada.

## Lo que esperamos aprender (en orden de prioridad)

**En qué secciones de `product-brief.md` importa realmente la especificidad.** Vamos a rastrear qué secciones del brief moldean más la UI generada: el planteamiento del problema, el usuario objetivo, la lista de funcionalidades MVP — y comparar los outputs cuando cualquiera de las tres es vaga versus específica.

**Si `ux-vibes.md` supera a los códigos hex.** Correremos una iteración solo con sitios de referencia y adjetivos de tono, y otra con hex codes y especificaciones tipográficas, para comparar cuál produce un resultado visual más coherente.

**Cómo maneja el agente la ambigüedad.** Anotaremos cada vez que el agente hace una pregunta de aclaración versus rellena con un default razonable — y si los defaults fueron aceptables.
