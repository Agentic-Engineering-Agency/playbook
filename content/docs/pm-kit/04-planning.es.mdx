# Fase 2 — Planeación

La planeación convierte la autorización del charter en un conjunto de compromisos concretos. Al final de esta fase, el equipo sabe _qué_ va a entregar (PRD, WBS, épicas e historias), _cuándo_ (cronograma con Gantt), _cuánto cuesta_ (estimación de costos), _qué puede salir mal_ (matriz de riesgo), _cómo fluye la información_ (plan de comunicaciones), _con qué estándar de calidad_ (plan de calidad) y _con qué equipo y presupuesto_ (plan de recursos).

Este es el momento en el que invertir tiempo produce el mayor retorno: un plan incompleto genera deuda de decisión que el equipo paga con intereses durante la ejecución.

## Cuándo comienza esta fase

La planeación comienza una vez que el charter está firmado y el registro de stakeholders tiene su primera versión aprobada. No es necesario que todos los artefactos de planeación se produzcan en el mismo orden para todos los proyectos, pero el PRD precede al WBS, el WBS precede al cronograma, y las épicas e historias alimentan tanto al WBS como al cronograma.

<h3 id="wbs">Artefacto 3: WBS — Estructura de Desglose del Trabajo</h3>

| | |
|---|---|
| **Skill** | `wbs` |
| **Tipo** | Extension — originalmente desarrollado para Agentic PM Kit |
| **Cuándo usarlo** | El backlog de épicas e historias está revisado y el equipo necesita una vista orientada a entregables del alcance — en lugar de una vista orientada a funcionalidades — antes de construir el cronograma o asignar recursos. |
| **Lista de aceptación** | `docs/pm-kit/checklists/wbs.md` dentro de tu proyecto instalado |

**Patrón de invocación.**

> _Invoca el skill `wbs`._

El agente construirá la WBS en tres niveles: fases o entregables principales (Nivel 1), sub-entregables (Nivel 2) y paquetes de trabajo (Nivel 3). Producirá un diagrama Mermaid `graph TD` con los nodos de Nivel 1 y Nivel 2 más una muestra representativa del Nivel 3, una tabla de detalle de paquetes de trabajo con código WBS, propietario, duración estimada y dependencias, y un diccionario WBS para los paquetes más críticos o ambiguos. El artefacto se guarda en `docs/pm-kit/outputs/wbs/`.

<h3 id="schedule-gantt">Artefacto 4: Cronograma / Gantt</h3>

| | |
|---|---|
| **Skill** | `schedule-gantt` |
| **Tipo** | Extension — originalmente desarrollado para Agentic PM Kit |
| **Cuándo usarlo** | La WBS está completa y las estimaciones de esfuerzo existen; el equipo necesita un cronograma base con dependencias y ruta crítica identificada, en formato de diagrama Gantt. |
| **Lista de aceptación** | `docs/pm-kit/checklists/schedule-gantt.md` dentro de tu proyecto instalado |

**Patrón de invocación.**

> _Invoca el skill `schedule-gantt`._

El agente asignará IDs de tarea, mapeará dependencias, calculará la ruta crítica, identificará hitos clave y producirá un bloque `gantt` de Mermaid con secciones alineadas a la WBS. Completará la tabla de detalle de tareas, la narrativa de ruta crítica y los supuestos y riesgos del cronograma. El artefacto se guarda en `docs/pm-kit/outputs/schedule-gantt/`.

<h3 id="cost-estimation">Artefacto 5: Estimación de Costos</h3>

| | |
|---|---|
| **Skill** | `cost-estimation` |
| **Tipo** | Skill liderado por el equipo — el agente enseña y facilita; el equipo estima |
| **Cuándo usarlo** | El backlog tiene historias o tareas sin dimensionar y el sprint planning es inminente, o se requiere una estimación total documentada con rango de confianza para una puerta de planeación. |
| **Lista de aceptación** | `docs/pm-kit/checklists/cost-estimation.md` dentro de tu proyecto instalado |

**Patrón de invocación.**

> _Invoca el skill `cost-estimation`._

El agente enseñará tres técnicas complementarias: **Planning Poker** (para historias de usuario con acuerdo de equipo, usando la secuencia de Fibonacci), **talla de camiseta** (para épicas o funcionalidades donde la precisión numérica no es todavía pertinente: S / M / L / XL) y **estimación de tres puntos / PERT** (para tareas donde la incertidumbre justifica un rango probabilístico, con fórmula `E = (O + 4M + P) / 6`). El agente facilita cada ronda y registra los valores producidos por el equipo — no fabrica cifras. El artefacto se guarda en `docs/pm-kit/outputs/cost-estimation/`.

<h3 id="risk-matrix">Artefacto 6: Matriz de Riesgo</h3>

| | |
|---|---|
| **Skill** | `risk-matrix` |
| **Tipo** | Extension — originalmente desarrollado para Agentic PM Kit |
| **Cuándo usarlo** | Una puerta de planeación o revisión de hito requiere un registro formal de riesgos antes de la aprobación; el charter está firmado y el equipo debe aflorar riesgos antes de que comience la ejecución. |
| **Lista de aceptación** | `docs/pm-kit/checklists/risk-matrix.md` dentro de tu proyecto instalado |

**Patrón de invocación.**

> _Invoca el skill `risk-matrix`._

El agente invertirá los criterios de éxito del proyecto para generar modos de falla, tensionará el plan en cuatro categorías (técnica, externa, organizacional, gestión de proyecto), consolidará candidatos en forma de "Si `<causa>` ocurre, entonces `<efecto>` puede impactar `<objetivo>`", puntuará cada riesgo en probabilidad × impacto (1–5 cada eje), asignará una estrategia de respuesta PMBOK (evitar / mitigar / transferir / aceptar) y producirá la lista de los cinco riesgos de mayor prioridad con planes de respuesta concretos. El artefacto se guarda en `docs/pm-kit/outputs/risk-matrix/`.

## Ejemplo trabajado — BookSwap Campus

### PRD — Documento de Requisitos del Producto

El PRD de BookSwap Campus v1 se produjo con el skill `prd` a partir del charter y el registro de interesados, y define 17 requisitos funcionales (FR-001 a FR-017) organizados alrededor de cuatro tipos de usuario: comprador estudiante, vendedor estudiante, administrador universitario y moderador de plataforma. Los requisitos no funcionales establecen cumplimiento de la LFPDPPP, accesibilidad WCAG 2.1 AA y un tiempo de respuesta p95 menor a 2 segundos — cifras que el equipo fijó antes de escribir una sola línea de código. Se registran cinco preguntas abiertas con dueño y fecha límite de resolución, incluida la situación legal del fideicomiso de pagos bajo regulación mexicana. [Ver PRD completo](/pm-kit/outputs/prd/bookswap-campus-v1.md)

### EDT — Estructura de Desglose del Trabajo

La WBS de BookSwap Campus v1 se produjo con el skill `wbs` y estructura el proyecto en 7 fases en el Nivel 1 (Fundación a Cierre), aproximadamente 32 paquetes de trabajo en el Nivel 3, y un diagrama `graph TD` de Mermaid que mapea fases con sub-entregables. Cada paquete lleva código WBS, responsable, duración estimada y dependencias; el diccionario WBS explica los cinco paquetes más ambiguos — incluyendo la integración del fideicomiso y la verificación del correo institucional — cuyos criterios de aceptación no eran claros en el PRD. La WBS se convirtió en el insumo directo para el cronograma y para la descomposición del backlog por sprint. [Ver WBS completa](/pm-kit/outputs/wbs/bookswap-campus-v1.md)

### Épicas e Historias

El artefacto de épicas e historias de BookSwap Campus v1 se produjo con el skill `epics-and-stories` y descompone el PRD en 7 épicas (EP-01 a EP-07) y 18 historias de usuario (ST-01 a ST-18). La distribución de prioridad es 12 Debe / 5 Debería / 1 Podría; cada historia incluye criterios de aceptación en notación Gherkin (Dado / Cuando / Entonces) y dependencias explícitas en historias previas, con lo que el equipo entró al sprint planning con un backlog ya secuenciado. EP-03 (Onboarding verificado) y EP-04 (Núcleo del mercado) concentran el mayor número de historias y mapean directamente a los riesgos R1 y R2 en zona roja de la matriz. [Ver épicas e historias completas](/pm-kit/outputs/epics-and-stories/bookswap-campus-v1.md)

### Cronograma (Gantt)

El cronograma de BookSwap Campus v1 se produjo con el skill `schedule-gantt` y abarca aproximadamente 160 días hábiles en la ruta crítica, desde el arranque el 2026-05-04 (M0) hasta el cierre formal el 2026-12-18 (M6), con el go-live en M4 = 2026-08-12. El bloque `gantt` de Mermaid define 7 hitos e identifica ramas paralelas — diseño UX e infraestructura corren en paralelo con el desarrollo de sprints para preservar holgura — y la narrativa señala las cinco tareas con holgura cero que retrasarían el go-live si se deslizan. El cronograma está cruzado con la WBS para que cualquier deslizamiento en un paquete de trabajo se propague visiblemente al hito correspondiente. [Ver cronograma completo](/pm-kit/outputs/schedule-gantt/bookswap-campus-v1.md)

### Estimación de costo — Sprint 4

La estimación de costo del Sprint 4 se facilitó con el skill `cost-estimation` y combina dos técnicas: Planning Poker (Fibonacci) para las 10 historias de usuario y estimación de tres puntos PERT para las 2 tareas de infraestructura. Planning Poker arrojó 68 puntos de historia en total; contra la velocidad observada de ~40 pts/sprint del Sprint 3, el backlog requiere ~1.7 sprints — así que el equipo comprometió un techo de 28 pts para el Sprint 4 y planificó arrastrar ~28 pts al Sprint 5, sujeto al supuesto del SDK del proveedor de pagos. Las dos tareas PERT produjeron una duración esperada combinada de 9.17 días hábiles con un intervalo de confianza al 80 % de [6.6 — 11.8 días]. [Ver estimación completa](/pm-kit/outputs/cost-estimation/sprint-4-backlog.md)

### Matriz de riesgos

La matriz de riesgos base (v1.0) se generó con la skill `risk-matrix` e identifica 12 riesgos distribuidos en las cuatro categorías PMBOK (técnica, externa, organizacional, gestión de proyecto). La fotografía inicial pone cuatro riesgos en zona roja: R1 adopción por debajo del umbral crítico y R2 fricción de onboarding —ambos con score 20— seguidos de R6 arquitectura ante el pico de inicio de ciclo y R8 anuncios con material bajo derecho de autor —ambos score 15—. Cada riesgo del Top-5 lleva un plan de respuesta con acciones concretas, dueño, condiciones de disparo y fecha objetivo (p. ej., R2 asignado al Product Owner con fecha fin del Sprint 4, 2026-05-08). El artefacto confirma el cruce de referencias con el charter: R2 protege la métrica "tiempo mediano a primera transacción ≤ 7 días" y R1 protege el objetivo de 1,500 usuarios al cierre del semestre. [Ver matriz completa](/pm-kit/outputs/risk-matrix/bookswap-campus-v1-planning.md)

### Plan de comunicaciones

El plan de comunicaciones de BookSwap Campus v1 se produjo con el skill `communication-plan` y mapea 11 audiencias distintas — desde la patrocinadora universitaria hasta los contactos técnicos del proveedor de pagos — a 8 formatos de reunión recurrente y 4 rutas de escalación. El plan documenta quién envía qué a quién, con qué cadencia y por qué canal, cubriendo desde el standup diario hasta el reporte trimestral al comité directivo de La Salle y Panamericana. Se incluye una política de retención documental de 2 años y reglas claras sobre cuándo escalar un bloqueo al patrocinador (sin resolución después de 2 días hábiles a nivel de equipo). [Ver plan de comunicaciones completo](/pm-kit/outputs/communication-plan/bookswap-campus-v1.md)

### Plan de calidad

El plan de calidad de BookSwap Campus v1 se produjo con el skill `quality-plan` y define 9 métricas de calidad medibles con metas mínimas obligatorias, incluyendo tiempo de respuesta p95 < 2 s, cobertura de pruebas automatizadas ≥ 70 %, cumplimiento WCAG 2.1 AA en todas las pantallas públicas y superación del OWASP Top 10 confirmada mediante prueba de penetración antes del M3. La calidad de proceso se garantiza mediante la disciplina de cinco pasos de SpecSafe — spec, pruebas, implementación, verificación, archivo — de modo que cada historia entra al sprint con criterios de aceptación y sale con pruebas que pasan. Una auditoría de accesibilidad y una prueba de penetración están programadas como puertas formales antes del go-live. [Ver plan de calidad completo](/pm-kit/outputs/quality-plan/bookswap-campus-v1.md)

### Plan de recursos

El plan de recursos de BookSwap Campus v1 se produjo con el skill `resource-plan` y conforma un equipo núcleo de 5 personas (PM, PO, Eng Lead, desarrollador backend, desarrollador frontend/UX) más 4 moderadores estudiantes a honorarios para el semestre piloto, con 4 roles Por definir cuyos plazos de designación se monitorean antes del M1. El costo total proyectado es MXN 1,357,000 — MXN 980,000 en personal y MXN 377,000 en infraestructura, licencias de software y contingencia — situando el proyecto dentro del envelope MXN 1.25M–1.75M autorizado en el charter. Las contrataciones pendientes (dos enlaces universitarios en La Salle y Panamericana) están señaladas con una fecha límite firme para no deslizar el hito de go-live. [Ver plan de recursos completo](/pm-kit/outputs/resource-plan/bookswap-campus-v1.md)

<Callout type="tip">
**Lo que no debe faltar en la planeación**

- **El orden de los artefactos importa.** El PRD alimenta las épicas e historias; las épicas e historias alimentan el WBS; el WBS alimenta el cronograma; el cronograma y el WBS alimentan la estimación de costos y el plan de recursos. Saltarte un eslabón crea inconsistencias que se vuelven costosas de corregir en ejecución.
- **La estimación de costos la hace el equipo, no el agente.** El skill de estimación enseña las técnicas y facilita las rondas, pero los números los produce el equipo. Un agente que "estima solo" produce cifras sin responsable — y sin responsable no hay compromiso.
- **La matriz de riesgo no es un trámite.** El ejercicio de inversión de metas y el tensionado en cuatro categorías está diseñado para aflorar los riesgos que el equipo _no_ ve todavía. Si la matriz sólo tiene los riesgos obvios, el ejercicio no terminó.
</Callout>
