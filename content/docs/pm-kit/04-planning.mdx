# Phase 2 — Planning

Planning converts the charter authorization into a set of concrete commitments. By the end of this phase, the team knows _what_ it will deliver (PRD, WBS, epics and stories), _when_ (schedule with Gantt), _how much it costs_ (cost estimation), _what can go wrong_ (risk matrix), _how information flows_ (communications plan), _to what quality standard_ (quality plan), and _with what team and budget_ (resource plan).

This is the moment where investing time yields the highest return: an incomplete plan generates decision debt that the team pays with interest during execution.

## When this phase begins

Planning begins once the charter is signed and the stakeholder register has its first approved version. Not every planning artifact must be produced in the same order for all projects, but the PRD precedes the WBS, the WBS precedes the schedule, and epics and stories feed both the WBS and the schedule.

<h3 id="prd">Artifact 1: PRD — Product Requirements Document</h3>

| | |
|---|---|
| **Skill** | `prd` |
| **Type** | Adapted from open-source MIT content. See `THIRD_PARTY_NOTICES.md`. |
| **When to use** | The sponsor has approved the charter and the team must now define functional and non-functional requirements before decomposing scope into a WBS or story backlog. |
| **Acceptance checklist** | `docs/pm-kit/checklists/prd.md` inside your installed project |

**Invocation pattern.**

> _Invoke the `prd` skill._

The agent will build the PRD in ten steps: overview, goals and non-goals, target users and use cases, functional requirements (numbered FR-01, FR-02 …), non-functional requirements, constraints and dependencies, success metrics, and open questions. The artifact is saved to `docs/pm-kit/outputs/prd/`.

<h3 id="epics-and-stories">Artifact 2: Epics and Stories</h3>

| | |
|---|---|
| **Skill** | `epics-and-stories` |
| **Type** | Adapted from open-source MIT content. See `THIRD_PARTY_NOTICES.md`. |
| **When to use** | The PRD has been reviewed and baselined, and the team needs an actionable story backlog with acceptance criteria before sprint planning or WBS construction. |
| **Acceptance checklist** | `docs/pm-kit/checklists/epics-and-stories.md` inside your installed project |

**Invocation pattern.**

> _Invoke the `epics-and-stories` skill._

The agent will group functional requirements into epics, decompose each epic into user stories in the format "As a `<user type>`, I want `<action>`, so that `<outcome>`", add acceptance criteria in Gherkin notation (Given / When / Then), assign priority (Must / Should / Could), and tag dependencies between stories. The artifact is saved to `docs/pm-kit/outputs/epics-and-stories/`.

<h3 id="wbs">Artifact 3: WBS — Work Breakdown Structure</h3>

| | |
|---|---|
| **Skill** | `wbs` |
| **Type** | Extension — originally authored for Agentic PM Kit |
| **When to use** | The epics-and-stories backlog is reviewed and the team needs a deliverable-oriented view of scope — rather than a feature-oriented view — before building the schedule or assigning resources. |
| **Acceptance checklist** | `docs/pm-kit/checklists/wbs.md` inside your installed project |

**Invocation pattern.**

> _Invoke the `wbs` skill._

The agent will construct the WBS across three levels: phases or major deliverables (Level 1), sub-deliverables (Level 2), and work packages (Level 3). It will produce a Mermaid `graph TD` diagram with Level 1 and Level 2 nodes plus a representative Level 3 sample, a work-package detail table with WBS code, owner, estimated duration, and dependencies, and a WBS dictionary for the most critical or ambiguous packages. The artifact is saved to `docs/pm-kit/outputs/wbs/`.

<h3 id="cost-estimation">Artifact 5: Cost Estimation</h3>

| | |
|---|---|
| **Skill** | `cost-estimation` |
| **Type** | Human-led skill — the agent teaches and facilitates; the team estimates |
| **When to use** | The backlog contains unsized stories or tasks and sprint planning is imminent, or a documented total estimate with a confidence range is required for a planning gate. |
| **Acceptance checklist** | `docs/pm-kit/checklists/cost-estimation.md` inside your installed project |

**Invocation pattern.**

> _Invoke the `cost-estimation` skill._

The agent will teach three complementary techniques: **Planning Poker** (for user stories requiring team agreement, using the Fibonacci sequence), **T-shirt sizing** (for epics or features where numeric precision is not yet warranted: S / M / L / XL), and **three-point / PERT estimation** (for tasks where uncertainty warrants a probabilistic range, using the formula `E = (O + 4M + P) / 6`). The agent facilitates each round and records the values the team produces — it does not fabricate figures. The artifact is saved to `docs/pm-kit/outputs/cost-estimation/`.

<h3 id="risk-matrix">Artifact 6: Risk Matrix</h3>

| | |
|---|---|
| **Skill** | `risk-matrix` |
| **Type** | Extension — originally authored for Agentic PM Kit |
| **When to use** | A planning gate or milestone review requires a formal risk register before approval; the charter is signed and the team must surface risks before execution begins. |
| **Acceptance checklist** | `docs/pm-kit/checklists/risk-matrix.md` inside your installed project |

**Invocation pattern.**

> _Invoke the `risk-matrix` skill._

The agent will invert the project's success criteria to generate failure modes, stress-test the plan across four categories (technical, external, organizational, project management), consolidate candidates as "If `<cause>` occurs, then `<effect>` may impact `<objective>`", score each risk on probability × impact (1–5 per axis), assign a PMBOK response strategy (avoid / mitigate / transfer / accept), and produce the top-5 priority risks with concrete response plans. The artifact is saved to `docs/pm-kit/outputs/risk-matrix/`.

## Worked example — BookSwap Campus

### Product Requirements Document

The BookSwap Campus v1 PRD was produced with the `prd` skill from the charter and stakeholder register, and defines 17 functional requirements (FR-001 to FR-017) organized around four user types: buyer student, seller student, university administrator, and platform moderator. Non-functional requirements mandate LFPDPPP data-privacy compliance, WCAG 2.1 AA accessibility, and a p95 response time under 2 seconds — figures the team set before writing a single line of code. Five open questions are tracked with named owners and resolution deadlines, including the legal status of escrow under Mexican payment regulation. [See full PRD](/pm-kit/outputs/prd/bookswap-campus-v1.md)

### Work Breakdown Structure

The BookSwap Campus v1 WBS was produced with the `wbs` skill and structures the project into 7 phases at Level 1 (Foundation through Closure), roughly 32 work packages at Level 3, and a Mermaid `graph TD` diagram that maps phases to sub-deliverables. Each package carries a WBS code, owner, estimated duration, and dependencies; the WBS dictionary calls out the five most ambiguous packages — including the escrow integration and the institutional email verification — whose acceptance criteria were unclear in the PRD. The WBS became the direct input for the schedule and for sprint backlog decomposition. [See full WBS](/pm-kit/outputs/wbs/bookswap-campus-v1.md)

### Epics and Stories

The BookSwap Campus v1 epics-and-stories artifact was produced with the `epics-and-stories` skill and decomposes the PRD into 7 epics (EP-01 through EP-07) and 18 user stories (ST-01 through ST-18). Priority distribution is 12 Must / 5 Should / 1 Could; every story includes Gherkin acceptance criteria (Given / When / Then) and explicit dependencies on prior stories, so the team entered sprint planning with a backlog that was already sequenced. EP-03 (Verified Onboarding) and EP-04 (Marketplace Core) carry the highest story count and map directly to the red-zone risks R1 and R2 in the risk matrix. [See full epics and stories](/pm-kit/outputs/epics-and-stories/bookswap-campus-v1.md)

### Schedule (Gantt)

The BookSwap Campus v1 schedule was produced with the `schedule-gantt` skill and spans approximately 160 business days on the critical path, from kick-off on 2026-05-04 (M0) to formal close on 2026-12-18 (M6), with go-live at M4 = 2026-08-12. The Mermaid `gantt` block defines 7 milestones and identifies parallel branches — UX design and infrastructure setup run concurrently with sprint development to preserve float — and the narrative calls out the five tasks with zero float that would delay go-live if they slip. The schedule is cross-referenced to the WBS so any slippage in a work package cascades visibly to its milestone. [See full schedule](/pm-kit/outputs/schedule-gantt/bookswap-campus-v1.md)

### Cost Estimation — Sprint 4

The Sprint 4 cost estimation was facilitated with the `cost-estimation` skill and combines two techniques: Planning Poker (Fibonacci) for the 10 user stories and PERT three-point estimation for the 2 infrastructure tasks. Planning Poker produced 68 total story points; against the team's observed velocity of ~40 pts/sprint from Sprint 3, the backlog requires ~1.7 sprints — so the team committed a 28-pt ceiling for Sprint 4 and planned to carry ~28 pts into Sprint 5 pending the payment-provider SDK assumption. The two PERT tasks produced a combined expected duration of 9.17 business days with an 80% confidence interval of [6.6 — 11.8 days]. [See full estimation](/pm-kit/outputs/cost-estimation/sprint-4-backlog.md)

### Risk Matrix

The baseline risk matrix (v1.0) was produced with the `risk-matrix` skill and identifies 12 risks across all four PMBOK categories (technical, external, organizational, project management). The initial snapshot places four risks in the red zone: R1 adoption below critical threshold and R2 onboarding friction (both scored 20), followed by R6 architecture under start-of-semester traffic and R8 listings containing copyrighted material (both scored 15). Every Top-5 risk carries a concrete response plan with actions, owner, trigger conditions, and target date (e.g., R2 is assigned to the Product Owner with a due date at end of Sprint 4, 2026-05-08). The artifact is cross-referenced with the charter: R2 guards the "median time-to-first-transaction ≤ 7 days" metric and R1 guards the 1,500-users end-of-semester objective. [See full matrix](/pm-kit/outputs/risk-matrix/bookswap-campus-v1-planning.md)

### Communications Plan

The BookSwap Campus v1 communications plan was produced with the `communication-plan` skill and maps 11 distinct audiences — from the university sponsor to payment provider technical contacts — to 8 recurring meeting formats and 4 escalation routes. The plan documents who sends what to whom, at what cadence, and through which channel, covering everything from the daily standup to the quarterly steering-committee report for La Salle and Panamericana. A 2-year document retention policy and clear rules about when to escalate a blocker to the sponsor (unresolved after 2 business days at team level) are included. [See full communications plan](/pm-kit/outputs/communication-plan/bookswap-campus-v1.md)

### Quality Plan

The BookSwap Campus v1 quality plan was produced with the `quality-plan` skill and defines 9 measurable quality metrics with hard minimum targets, including p95 response time < 2 s, automated test coverage ≥ 70%, WCAG 2.1 AA compliance for all public screens, and OWASP Top 10 clearance confirmed by a penetration test before M3. Process quality is enforced via the SpecSafe five-step discipline — spec, tests, implementation, verify, archive — so that every story enters the sprint with acceptance criteria and exits with tests that pass. A penetration test and accessibility audit are scheduled as formal gates before go-live. [See full quality plan](/pm-kit/outputs/quality-plan/bookswap-campus-v1.md)

### Resource Plan

The BookSwap Campus v1 resource plan was produced with the `resource-plan` skill and assembles a core team of 5 (PM, PO, Eng Lead, backend developer, frontend/UX developer) plus 4 honorario student moderators for the pilot semester, with 4 TBD roles whose designation deadlines are tracked before M1. Total projected cost is MXN 1,357,000 — MXN 980,000 in personnel and MXN 377,000 in infrastructure, software licenses, and contingency — placing the project within the MXN 1.25M–1.75M envelope authorized in the charter. Open hiring tasks (two university liaisons at La Salle and Panamericana) are flagged with a hard deadline to avoid slipping the go-live milestone. [See full resource plan](/pm-kit/outputs/resource-plan/bookswap-campus-v1.md)

<Callout type="tip">
**What planning must get right**

- **Artifact order matters.** The PRD feeds epics and stories; epics and stories feed the WBS; the WBS feeds the schedule; the schedule and WBS together feed cost estimation and the resource plan. Skipping a link creates inconsistencies that become expensive to fix in execution.
- **Cost estimation is done by the team, not the agent.** The cost-estimation skill teaches the techniques and facilitates the rounds, but the numbers come from the team. An agent that "estimates alone" produces figures with no owner — and without an owner there is no commitment.
- **The risk matrix is not a formality.** The goal-inversion exercise and the four-category stress test are designed to surface the risks the team does not yet see. If the matrix contains only the obvious risks, the exercise is not finished.
</Callout>
